Tell your hosting provider you're running a security research/training platform. Open a support ticket: "We're hosting a controlled cybersecurity training lab. Attacker traffic is intentional and contained within our isolated network. No external targets are involved." This prevents abuse reports.

Part 1B: Full Disk Encryption (LUKS)

Encrypt BEFORE storing any sensitive data. This protects against physical disk theft, data center breaches, and decommissioned drives retaining data. AES-256 (XTS mode) — the same standard used by classified government systems.
Two approaches depending on your provider:
Option A: Provider offers Kali/Debian install with LUKS (Hetzner installimage) — use it.
Option B: Provider pre-installs OS without encryption — encrypt the data partition after install.

Option A: Hetzner installimage with LUKS (Recommended)

A1

Boot into Hetzner Rescue System (from their dashboard: Rescue tab → Activate).

A2

SSH into rescue and run installimage with encryption:

rescue# installimage # In the config editor, set: DRIVE1 /dev/sda # or /dev/nvme0n1 SWRAID 0 # No software RAID (single disk) BOOTLOADER grub HOSTNAME dagda-colosseum PART /boot ext4 1G PART / ext4 all crypt # ← THIS enables LUKS encryption IMAGE /root/.oldroot/nfs/images/Debian-1210-bookworm-amd64-base.tar.gz # Save and confirm. It will ask for a LUKS passphrase. # Use 20+ characters. DIFFERENT from your user password. # Store this passphrase securely — losing it = losing the server.
A3

Set up remote LUKS unlock via SSH (so you can reboot without physical console):

# apt install -y dropbear-initramfs # Add your SSH public key for initramfs unlock # echo "YOUR_PUBLIC_KEY" > /etc/dropbear/initramfs/authorized_keys # Configure dropbear # echo 'DROPBEAR_OPTIONS="-p 2222 -s -j -k"' >> /etc/dropbear/initramfs/dropbear.conf # Rebuild initramfs # update-initramfs -u # After reboot, unlock remotely: local$ ssh -p 2222 root@YOUR_SERVER_IP initramfs$ cryptroot-unlock # Enter LUKS passphrase, server continues booting

Option B: Post-Install Data Partition Encryption

If your provider doesn't support LUKS during install, encrypt the data partition where Dagda stores its databases, logs, and session data:

B1

Create an encrypted partition for Dagda data:

# Create a LUKS-encrypted file container (if no spare partition) # dd if=/dev/urandom of=/root/dagda-vault.img bs=1M count=50000 # 50GB # cryptsetup luksFormat --type luks2 \ --cipher aes-xts-plain64 \ --key-size 512 \ --hash sha512 \ --iter-time 5000 \ /root/dagda-vault.img # Enter passphrase (20+ characters) # Open the encrypted volume # cryptsetup open /root/dagda-vault.img dagda-vault # Create filesystem # mkfs.ext4 /dev/mapper/dagda-vault # Mount # mkdir -p /opt/dagda-data # mount /dev/mapper/dagda-vault /opt/dagda-data # chown dagda:dagda /opt/dagda-data
B2

Configure Dagda to store data on the encrypted volume:

# In dagda.toml: [general] data_dir = "/opt/dagda-data" log_dir = "/opt/dagda-data/logs" [security] audit_log = "/opt/dagda-data/audit.log"
B3

Create a startup script to unlock the vault before Dagda starts:

$ sudo tee /usr/local/bin/dagda-unlock.sh << 'SCRIPT' #!/bin/bash if ! cryptsetup status dagda-vault &>/dev/null; then echo "Unlocking Dagda vault..." cryptsetup open /root/dagda-vault.img dagda-vault mount /dev/mapper/dagda-vault /opt/dagda-data fi SCRIPT $ sudo chmod 700 /usr/local/bin/dagda-unlock.sh
Encryption specs:
• Algorithm: AES-256-XTS (512-bit key, split into two 256-bit AES keys)
• Hash: SHA-512 for key derivation
• Key derivation: PBKDF2 with 5000ms iteration time (brute-force resistant)
• This matches FIPS 140-2 Level 1 and NATO RESTRICTED classification requirements

Part 2: Base OS Installation and Hardening

4

Create a non-root user and disable root SSH:

# adduser dagda # usermod -aG sudo dagda # Set up SSH key authentication # mkdir -p /home/dagda/.ssh # echo "YOUR_PUBLIC_KEY_HERE" > /home/dagda/.ssh/authorized_keys # chmod 700 /home/dagda/.ssh && chmod 600 /home/dagda/.ssh/authorized_keys # chown -R dagda:dagda /home/dagda/.ssh # Harden SSH # nano /etc/ssh/sshd_config # Set: PermitRootLogin no # Set: PasswordAuthentication no # Set: MaxAuthTries 3 # systemctl restart sshd
Test SSH key login BEFORE disabling password auth. Open a new terminal, verify ssh dagda@YOUR_SERVER_IP works with your key. If it doesn't, you'll lock yourself out.
5

Configure the firewall:

$ sudo apt install -y ufw $ sudo ufw default deny incoming $ sudo ufw default allow outgoing $ sudo ufw allow 22/tcp comment "SSH" $ sudo ufw allow 443/tcp comment "Colosseum HTTPS" $ sudo ufw enable $ sudo ufw status
6

Apply kernel hardening:

$ sudo tee /etc/sysctl.d/99-colosseum.conf << 'EOF' net.ipv4.tcp_syncookies = 1 net.ipv4.conf.all.rp_filter = 1 net.ipv4.conf.all.accept_redirects = 0 net.ipv4.conf.all.send_redirects = 0 net.ipv6.conf.all.accept_redirects = 0 kernel.randomize_va_space = 2 kernel.kptr_restrict = 2 kernel.dmesg_restrict = 1 net.ipv4.ip_forward = 1 # Needed for container networking net.ipv4.conf.all.forwarding = 1 # Needed for VM networking EOF $ sudo sysctl --system
7

Update everything and install prerequisites:

$ sudo apt update && sudo apt full-upgrade -y $ sudo apt install -y \ python3-pip python3-venv git curl wget jq \ podman podman-compose \ qemu-kvm libvirt-daemon-system virtinst \ nginx certbot python3-certbot-nginx \ nmap suricata zeek \ auditd fail2ban

$ curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh $ source ~/.cargo/env $ cd /opt/dagda/core $ cargo build --release $ pip install maturin $ maturin develop --release $ cd /opt/dagda

$ curl -fsSL https://ollama.com/install.sh | sh $ ollama pull llama3.2:3b

Part 4: Configure Container Infrastructure

13

Configure Podman for rootless containers (more secure):

$ sudo loginctl enable-linger dagda $ podman info | head -5 # Should show: host: ... rootless: true
14

Create the isolated Colosseum network:

# Create an internal-only Podman network (no internet access) $ podman network create \ --subnet 10.99.0.0/24 \ --gateway 10.99.0.1 \ --internal \ colosseum-arena # Verify it's internal (no external routing) $ podman network inspect colosseum-arena | jq '.[] | .internal' # Should output: true

Part 5: Build the Attacker Container Image

15

Create the Dockerfile for attacker containers:

$ mkdir -p /opt/dagda/colosseum/docker $ cat > /opt/dagda/colosseum/docker/Dockerfile.attacker << 'EOF' FROM kalilinux/kali-rolling:latest RUN apt-get update && apt-get install -y --no-install-recommends \ nmap nikto gobuster dirb sqlmap hydra john hashcat \ crackmapexec impacket-scripts enum4linux smbclient \ netcat-openbsd socat curl wget python3 python3-pip \ vim nano less tmux git openssh-client whois dnsutils \ wordlists seclists \ && apt-get clean && rm -rf /var/lib/apt/lists/* # Create non-root attacker user RUN useradd -m -s /bin/bash attacker USER attacker WORKDIR /home/attacker # Welcome message RUN echo 'echo "=== DAGDA COLOSSEUM ==="' >> ~/.bashrc && \ echo 'echo "Target network: 10.99.0.0/24"' >> ~/.bashrc && \ echo 'echo "Objective: shown in the web UI"' >> ~/.bashrc && \ echo 'echo "Time limit: shown in the web UI"' >> ~/.bashrc && \ echo 'echo ""' >> ~/.bashrc CMD ["/bin/bash"] EOF
16

Build the attacker image:

$ cd /opt/dagda/colosseum/docker $ podman build -t dagda-colosseum-kali -f Dockerfile.attacker . # This takes 5-10 minutes (downloading Kali packages) # Verify $ podman images | grep colosseum # dagda-colosseum-kali latest ... ~2GB
17

Test the attacker container:

# Spin up a test container on the isolated network $ podman run --rm -it \ --name test-attacker \ --network colosseum-arena \ --cpus 2 --memory 2g \ dagda-colosseum-kali # Inside the container: attacker$ nmap --version # Should work attacker$ ping 8.8.8.8 # Should FAIL (no internet) attacker$ exit

Part 7: Configure Network Isolation

This is the most critical section. Incorrect isolation means attackers can escape the arena and access your server or the internet.
21

Verify container network isolation:

# Start a test attacker container $ podman run --rm -it --network colosseum-arena dagda-colosseum-kali # These should SUCCEED (target VMs accessible): attacker$ ping -c1 10.99.0.10 # ✓ Target web server attacker$ nmap -sn 10.99.0.0/24 # ✓ Target network # These should FAIL (isolation working): attacker$ ping -c1 8.8.8.8 # ✗ No internet attacker$ ping -c1 YOUR_SERVER_IP # ✗ No host access attacker$ curl https://google.com # ✗ No internet attacker$ exit
22

Add iptables rules to prevent container escape:

# Block containers from reaching the host's management interface $ sudo iptables -I FORWARD -s 10.99.0.0/24 -d YOUR_SERVER_IP -j DROP $ sudo iptables -I FORWARD -s 10.99.0.0/24 ! -d 10.99.0.0/24 -j DROP # Save rules to persist across reboots $ sudo apt install -y iptables-persistent $ sudo netfilter-persistent save
23

Connect Dagda to the arena network for monitoring:

# Dagda needs to see traffic on the arena network # Option A: Run Dagda in a container on the same network # Option B: Use eBPF on the host to monitor the bridge interface # Find the bridge interface $ ip link | grep -E "virbr-arena|cni-podman" # Note the interface name (e.g., cni-podman1) # Dagda will monitor this interface via eBPF and pcap tools

Part 8: TLS and Domain Setup

24

Point a domain to your server. In your DNS provider:

# A record: colosseum.yourdomain.com → YOUR_SERVER_IP # Wait for DNS propagation (5-30 minutes) $ dig colosseum.yourdomain.com # Should resolve to your server IP
25

Get a TLS certificate with Let's Encrypt:

$ sudo certbot certonly --standalone \ -d colosseum.yourdomain.com \ --email your-email@example.com \ --agree-tos --non-interactive # Certificate saved to: # /etc/letsencrypt/live/colosseum.yourdomain.com/fullchain.pem # /etc/letsencrypt/live/colosseum.yourdomain.com/privkey.pem
26

Configure Nginx as reverse proxy:

$ sudo tee /etc/nginx/sites-available/colosseum << 'EOF' server { listen 443 ssl http2; server_name colosseum.yourdomain.com; ssl_certificate /etc/letsencrypt/live/colosseum.yourdomain.com/fullchain.pem; ssl_certificate_key /etc/letsencrypt/live/colosseum.yourdomain.com/privkey.pem; ssl_protocols TLSv1.2 TLSv1.3; ssl_ciphers HIGH:!aNULL:!MD5; # Colosseum web UI location / { proxy_pass http://127.0.0.1:8443; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_set_header X-Forwarded-Proto $scheme; } # WebSocket for terminal location /colosseum/ws/ { proxy_pass http://127.0.0.1:8443; proxy_http_version 1.1; proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection "upgrade"; proxy_set_header Host $host; proxy_read_timeout 3600s; } } server { listen 80; server_name colosseum.yourdomain.com; return 301 https://$host$request_uri; } EOF $ sudo ln -sf /etc/nginx/sites-available/colosseum /etc/nginx/sites-enabled/ $ sudo nginx -t && sudo systemctl reload nginx

Part 9: Launch the Colosseum

27

Create a systemd service for persistent operation:

$ sudo tee /etc/systemd/system/dagda-colosseum.service << 'EOF' [Unit] Description=Dagda Colosseum — Live-Fire Arena After=network.target podman.service libvirtd.service Wants=podman.service [Service] Type=simple User=dagda WorkingDirectory=/opt/dagda Environment=PATH=/opt/dagda/.venv/bin:/usr/local/bin:/usr/bin:/bin ExecStart=/opt/dagda/.venv/bin/python -m dagda --mode colosseum Restart=on-failure RestartSec=10 [Install] WantedBy=multi-user.target EOF $ sudo systemctl daemon-reload $ sudo systemctl enable dagda-colosseum $ sudo systemctl start dagda-colosseum
28

Verify the Colosseum is running:

$ sudo systemctl status dagda-colosseum # Should show: active (running) $ curl -k https://localhost:8443/colosseum/stats # Should return JSON: {"active_sessions": 0, "total_attackers": 0, ...} # Open in browser: # https://colosseum.yourdomain.com
The Colosseum is live. Attackers can register at your domain, choose a scenario, and get a browser terminal connected to your isolated target network. Dagda defends autonomously. Everything is logged.

Monthly

# Full system update $ sudo apt update && sudo apt full-upgrade -y # Rebuild attacker image (get latest tools) $ cd /opt/dagda/colosseum/docker $ podman build --no-cache -t dagda-colosseum-kali -f Dockerfile.attacker . # Review Dagda's self-audit $ python -c " from brain.self_security import DagdaSelfAudit a = DagdaSelfAudit() r = a.run() print(f'{r[\"summary\"][\"passed\"]} passed, {r[\"summary\"][\"failed\"]} failed') "

Verification Checklist

Complete every item before opening to public attackers:

Isolation

  • Attacker container CANNOT ping 8.8.8.8 (no internet)
  • Attacker container CANNOT ping host server IP
  • Attacker container CAN ping target VMs (10.99.0.x)
  • iptables FORWARD rules block 10.99.0.0/24 → outside
  • Podman network is marked as "internal"

Security

  • SSH root login disabled
  • SSH password auth disabled (key only)
  • UFW firewall active (only 22 + 443 open)
  • TLS certificate valid and auto-renewing
  • Dagda integrity baseline created
  • PANIC button works (Ctrl+Shift+P on TUI session)

Functionality

  • Colosseum web UI loads at https://your-domain
  • Registration + login works
  • Scenario cards display correctly
  • Terminal connects and provides working shell
  • Tools work inside container (nmap, sqlmap, etc.)
  • Dagda detects scans from attacker container
  • Session timer counts down correctly
  • Session end calculates score
  • Leaderboard updates after session

Resilience

  • Dagda restarts automatically after reboot (systemd)
  • Containers are destroyed when sessions end
  • Orphaned containers are cleaned up
  • Server survives attacker DoS attempt inside container (resource limits work)

Troubleshooting

Attacker can reach the internet

# Verify Podman network is internal $ podman network inspect colosseum-arena | jq '.[] | .internal' # Must be: true # If false, recreate the network: $ podman network rm colosseum-arena $ podman network create --subnet 10.99.0.0/24 --internal colosseum-arena # Verify iptables rules: $ sudo iptables -L FORWARD -n | grep 10.99

Terminal doesn't connect

# Check Nginx WebSocket config $ sudo nginx -t # Check Dagda is listening $ ss -tlnp | grep 8443 # Check container is running $ podman ps | grep colosseum

Dagda not detecting attacks

# Verify Dagda is monitoring the arena network interface $ ip link | grep -E "virbr|cni" # Dagda should be watching this interface # Check eBPF is available $ python -c "from tools.defensive.ebpf_monitor import EbpfMonitor; print(EbpfMonitor.is_available())" # Should be: True (on Linux as root)

Emergency shutdown

# Kill everything immediately $ sudo systemctl stop dagda-colosseum $ podman stop --all $ podman rm --all --force # Or from Dagda TUI: Ctrl+Shift+P (PANIC)

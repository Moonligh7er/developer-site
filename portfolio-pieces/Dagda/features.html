Core System (5)

5-Tier Safeguard System

FORTRESS (5) > VIGILANT (4) > BALANCED (3) > UNLOCKED (2) > ARENA (1). Higher = more restrictive. Controls what the AI can do autonomously. Raise freely; lowering requires confirmation phrases. FORTRESS requires 2FA (TOTP) to leave.

9 Operating Modes

SENTINEL (monitoring/defense/hardening) > TRIAGE (alert assessment) > INVESTIGATOR (OSINT/forensics) > STEALTH INVESTIGATOR (covert evidence) > DECEPTION (honeypots/canary tokens/counter-intel) > INCIDENT COMMANDER (IR case management) > BUG HUNTER (vuln research) > ARENA (purple team) > SELF-DEFENSE (IR/hardening/counter-intel). Each mode unlocks at specific safeguard levels.

Hash-Chained Audit Log

Every action is logged in an append-only, SHA-256 hash-chained JSON Lines file. Each entry links to the previous via cryptographic hash. Any tampering breaks the chain and is detectable on startup.

Permission Gate

Capability-based access control. Every tool call is checked against current mode, safeguard level, target scope, rate limits, and confirmation requirements before execution.

Unbreakable Guardrails

8 hard-coded rules that cannot be overridden: no data exfiltration, no audit tampering, no self-modification, no out-of-scope targeting, no unlogged execution, no fake capabilities, no stealth persistence, always explain autonomous reasoning.

AI Safeguards (14)

Philosophy: Safety by Design

Dagda is built on the principle that powerful security tools demand equally powerful safety constraints. Every capability has a corresponding control. Offensive features are locked by default and require deliberate, auditable escalation to unlock. The AI cannot override its own restrictions, cannot suppress logging, and cannot operate outside defined scope — ever. Dagda proves that a security tool can be both potent and responsible.

5-Tier Safeguard Hierarchy

Five levels from FORTRESS (5, maximum restriction) to ARENA (1, full offensive in isolated VMs). Default is BALANCED (3): safe operations run autonomously, offensive tools are locked. Raising the safeguard level is always instant and unrestricted. Lowering requires typed confirmation phrases proportional to the risk — the more you unlock, the more deliberately you must act. FORTRESS requires TOTP 2FA to leave, preventing social engineering attacks against the operator.

8 Unbreakable Guardrails

Hard-coded rules enforced at the permission gate level, not in the AI prompt. These cannot be overridden by any user, mode, or safeguard level: (1) No data exfiltration — tools cannot send data to unauthorized destinations. (2) No audit tampering — the hash-chained log is append-only and cryptographically verified. (3) No self-modification — the AI cannot alter its own code, config, or constraints. (4) No out-of-scope targeting — offensive tools only work against user-defined scope whitelist. (5) No unlogged execution — every tool call is recorded before it runs. (6) No fake capabilities — the AI cannot claim tools or permissions it doesn't have. (7) No stealth persistence — the AI cannot install backdoors, scheduled tasks, or hidden processes. (8) Always explain autonomous reasoning — when acting without explicit instruction, the AI must log its reasoning chain.

Permission Gate (Pre-Execution Enforcement)

Every tool invocation passes through a capability-based permission gate before execution. The gate checks: (a) Is the tool allowed in the current operating mode? (b) Does the current safeguard level meet or exceed the tool's minimum? (c) Is the target within the authorized scope whitelist? (d) Has the per-tool rate limit been exceeded? (e) Does this tool require user confirmation, and has it been granted? (f) Is the AI operating autonomously, and is autonomous execution allowed for this tool? Any check failure blocks execution and logs the denial. The gate is implemented in Rust (with Python fallback) for tamper resistance.

Hash-Chained Audit Trail

Every action, tool call, permission decision, and mode change is logged to an append-only JSON Lines file where each entry contains a SHA-256 hash of the previous entry. This creates a cryptographic chain — if any entry is modified, deleted, or reordered, the chain breaks and the tampering is detected on the next startup integrity check. The audit log cannot be disabled, paused, or redirected. Even audit log reads are logged.

Scope-Locked Offensive Operations

All offensive and reconnaissance tools are scope-locked: they can only target IP addresses, domains, and network ranges explicitly whitelisted by the user. Scope is defined before entering BUG HUNTER or ARENA mode and enforced by the permission gate at execution time — not by the AI's judgment. Attempting to target anything outside scope is blocked, logged, and triggers an alert. This prevents accidental or adversarial misuse against unauthorized systems.

Confirmation Escalation

Dangerous operations require explicit user confirmation even when the AI is operating autonomously. Confirmation requirements are tiered: (1) Low-risk defensive tools run without prompting. (2) Medium-risk tools (active scanning, network changes) require a single confirmation. (3) High-risk tools (exploit execution, system modification) require confirmation with a typed phrase. (4) Safeguard de-escalation requires progressively stronger confirmation — from a simple 'yes' down to TOTP codes. The AI cannot auto-confirm on the user's behalf.

Mode Isolation

Each operating mode (SENTINEL, TRIAGE, INVESTIGATOR, STEALTH INVESTIGATOR, DECEPTION, INCIDENT COMMANDER, BUG HUNTER, ARENA, SELF-DEFENSE) has a strictly defined tool allowlist. Tools registered for one mode are invisible and uncallable from another. This means a monitoring session cannot accidentally invoke offensive tools, and an offensive session cannot access evidence vault tools. Mode transitions are logged and require appropriate safeguard levels.

Autonomous Action Transparency

When the AI takes action without explicit user instruction (e.g., automated threat response, self-defense), it must: (a) Log the reasoning chain that led to the decision. (b) Record the specific threat indicators that triggered the response. (c) Note which playbook or escalation chain was followed. (d) Make all autonomous actions visible in the TUI defense feed and audit log. The user can review, reverse, or adjust any autonomous action after the fact.

Rate Limiting & Anti-Runaway

Per-tool rate limits with configurable per-minute caps prevent the AI from executing runaway automation loops. If a tool is called too frequently, subsequent calls are blocked until the window resets. This prevents resource exhaustion, accidental DDoS against targets, and infinite retry loops. Rate limit violations are logged and surfaced as warnings.

Kill Switch (Ctrl+Shift+K)

Instant emergency stop accessible from any screen. Terminates all running tool executions, cancels pending autonomous actions, and raises the safeguard level to FORTRESS. The kill switch cannot be disabled, intercepted, or delayed by any tool, plugin, or AI action. It is the operator's absolute override.

Destructive Command Block List

50+ regex patterns blocking dangerous shell commands before they reach the OS. Covers recursive deletion (rm -rf /), disk wiping (dd of=/dev/sda), fork bombs, iptables flush, chmod 777 /, mkfs on live partitions, and other catastrophic operations. Applied at the shell_execute layer regardless of mode, safeguard level, or AI reasoning. Cannot be disabled or overridden.

Red Team Execute Parameter

AI-initiated command execution with mandatory human approval. When the orchestrator determines a shell command is needed (e.g., during bug hunting or incident response), the command is presented to the operator for explicit approval before execution. The operator sees the exact command, its classification, and risk level. No command runs without a human pressing "approve." All approved and denied commands are logged to the audit trail.

Defensive Tools (5)

System Info

Comprehensive system snapshot: OS, CPU (cores/MHz/usage), memory, swap, disk partitions, network interfaces, logged-in users, boot time, process count.

Process Monitor

List and filter running processes. Detect suspicious spawns from /tmp, /dev/shm, or /var/tmp. Flag deleted executables, high resource usage, and name/exe mismatches.

Network Monitor

Track active connections (TCP/UDP), listening ports with owning processes, and per-interface I/O stats (bytes, packets, errors, drops).

Log Analyzer

Parse system logs for security events: failed/successful logins, sudo usage, SSH events, firewall blocks, service changes, privilege escalation, suspicious commands. Configurable time windows and custom regex.

Hardening Assessment

30+ CIS-aligned security checks: file permissions on sensitive files, SSH config (root login, password auth, X11), kernel params (ASLR, SYN cookies, IP forwarding), user accounts (UID 0, empty passwords), dangerous services (telnet, rsh, FTP). Returns findings with remediation advice.

Monitoring & Defense (7)

Real-Time Monitor Engine

Background async engine checking CPU, memory, disk, network, and processes every 5 seconds. Configurable thresholds for warnings and critical alerts. Feeds live data to the system monitor dashboard.

Anomaly Detection

Detects: CPU/memory/disk spikes above thresholds, high process spawn rates (>50/min), new listening ports, resource exhaustion patterns.

Brute Force Detection

Tracks failed login attempts in a configurable sliding window (default: 5 failures in 60 seconds). Emits critical alert on threshold breach.

File Integrity Monitoring

AIDE-style monitoring of critical system files (/etc/passwd, /etc/shadow, /etc/sudoers, /etc/ssh/sshd_config, etc.). Detects content changes, permission modifications, and ownership changes via SHA-256 checksums.

Self-Defense Engine

Autonomous threat response: block IPs (nftables/iptables), kill suspicious processes, quarantine files. All actions are logged, reversible (with undo commands), and respect hard constraints (never kill systemd/sshd/init).

IDS/IPS Integration

Suricata EVE JSON log parser with real-time alert monitoring. MITRE ATT&CK technique mapping. Rule reload support. Generic IDS fallback for non-Suricata systems.

Container Security

Docker/Podman runtime auditing and Dockerfile static analysis. List running containers with security metadata. Deep-inspect containers for misconfigs (privileged, host network, capabilities, mounts). Batch audit all running containers. Dockerfile static analysis checks: USER root, no HEALTHCHECK, ADD vs COPY, curl|bash, :latest tags, secrets in ENV/ARG, --no-check-certificate. Container network exposure audit (port bindings, host-network). Environment variable and label scanning for embedded secrets. Full baseline report generation. Auto-detects docker vs podman runtime.

Network Defense (6)

Network Scanner Engine

Two-tier scanning: fast passive cycle (10s, reads ARP table + mDNS/SSDP) and deep active cycle (5min, scapy ARP scan + nmap fingerprinting). Root/CAP_NET_RAW auto-detection: full mode (passive + deep scans), passive-only mode (ARP table + psutil), or unavailable. Background scapy AsyncSniffer in promiscuous mode with BPF filter for ARP/DHCP/DNS/mDNS/SSDP.

Trusted Network Profiles

SQLite-backed trusted network profiles. Networks identified by SHA-256(SSID + BSSID + gateway_MAC). Save networks as trusted — scanner only activates for known networks, never public WiFi. Per-network device inventory persisted across sessions. Protection levels: minimal, standard, paranoid.

8 Passive Threat Detectors

ARP spoofing (gateway MAC mismatch), rogue DHCP (unauthorized DHCP server), port scanning (>15 ports/60s), DNS spoofing (unexpected resolver), new device alerts, anomalous traffic (>3x rolling average), MAC changes (known IP with different MAC), deauth attacks (>10 deauth frames/5s). MITRE ATT&CK mapped.

Non-Intrusive Defense Actions

Gratuitous ARP correction (broadcast correct gateway binding), ARP cache pinning (static gateway entry), MAC-level blocking (nftables/ebtables bridge rules). All Layer 2/3 — no agents or config changes on other devices. Full cleanup on engine stop.

Network Map Widget

Live TUI display of network topology: device list with trust indicators (T=trusted, ?=unknown, !=suspicious, X=blocked, G=gateway), device count, threat status, scanner engine state. Toggle with Ctrl+Shift+N.

30 Network Defense Tools

6 scanner tools (scan, list devices, device detail/history, topology), 8 network profile tools (trust/untrust networks, trust/untrust devices, protection levels), 6 threat tools (list/detail/stats, ARP/gateway/DNS integrity checks), 8 defense tools (block/unblock devices, ARP protection, engine lifecycle), 2 intel tools (attack surface, risk summary).

Investigation Tools (8)

OSINT Suite

Passive intelligence gathering: DNS lookups (A/AAAA/MX/NS/TXT/CNAME via DNS-over-HTTPS), WHOIS queries, IP geolocation and ASN info, passive subdomain enumeration via Certificate Transparency (crt.sh), HTTP security header analysis, and email domain validation. All techniques are passive — no active probing.

Forensic Analysis

Read-only file forensics: multi-algorithm hashing (MD5/SHA-1/SHA-256), detailed file metadata with magic number detection, printable string extraction from binaries (ASCII + UTF-16), hex dump viewer, filesystem timeline analysis, and YARA rule scanning (requires yara-python).

Evidence Chain of Custody

Full evidence management with SQLite-backed case database. Create investigation cases, collect evidence with automatic SHA-256 hashing, verify evidence integrity at any time, maintain timestamped chain-of-custody logs, add investigator notes, and export case reports in JSON or text format.

Threat Intelligence

IoC checking against free threat feeds: MalwareBazaar for file hash reputation, AbuseIPDB for IP reputation (optional key), IP geolocation via ip-api.com, MITRE ATT&CK technique lookup (local database of 40+ techniques), and STIX 2.1 bundle parsing. Graceful degradation when optional API keys are not configured.

Cross-Source Correlation

Ties together OSINT, forensics, and intel findings. Correlates indicators across all evidence in a case, builds chronological investigation timelines, discovers connections between indicators (shared infrastructure, common IoCs), and generates consolidated IoC lists for sharing.

Active Directory LDAP Enumeration

Read-only LDAP enumeration and security auditing of Active Directory domains. Enumerate users and groups with attributes, find Kerberoasting candidates (SPNs), detect accounts with unconstrained/constrained delegation privileges, extract domain password policies, identify admin accounts (Domain Admins, Enterprise Admins, Administrators), and export results as BloodHound JSON. Optional ldap3 dependency with graceful fallback. All discovered entities added to Knowledge Graph.

Cloud Security Analysis

Passive cloud security posture analysis (AWS/GCP/Azure) with no cloud SDKs. Detect cloud provider via IMDS metadata endpoints. Probe for exposed metadata (instance IDs, IAM roles). Scan directories for hardcoded credentials (AWS keys, GCP service accounts, Azure secrets, GitHub tokens). Analyze IAM policies for dangerous permissions (wildcards, missing conditions). Audit Terraform/CloudFormation for S3 public exposure, overly permissive security groups, unencrypted resources, publicly accessible databases. Generate comprehensive cloud security reports with risk scoring (0-100 scale).

Supply Chain Security

Software dependency analysis and supply chain risk assessment. Generate CycloneDX SBOMs from installed packages or requirements files. Scan packages against the OSV batch API for known CVEs. Check outdated versions via PyPI JSON API. Verify package hashes against PyPI release checksums for tamper detection. Audit lockfiles for version pinning, hash-pinning, and conflicts. Combined dep_audit for vuln + outdated + license checks. Overall supply chain risk scoring (vuln count, outdated ratio, maintainer health).

Intelligence Layer (8)

Hypothesis-Driven Investigation

Structured hypothesis management for investigations. Create hypotheses with Bayesian prior confidence, link as parent/child trees, attach supporting and contradicting evidence. Test hypotheses against the knowledge graph (entity queries), IoC correlation (cross-store search across incidents and attackers), or pattern matching (MITRE technique entities). Bayesian confidence updates chain likelihood ratios through all evidence to compute posterior probabilities. Tree view shows all hypotheses with confidence scores, evidence counts, and status. Auto-suggest generates new hypotheses from MITRE technique chain analysis (follow-on and predecessor techniques), knowledge graph entity relationships, and uninvestigated entity coverage gaps.

Attack Surface Mapping

Enumerate all listening services via psutil with internet-facing vs internal classification. 25+ well-known port metadata entries with service names, risk notes, and hardening advice. Entities (host, port, service) automatically added to the knowledge graph. Analyze attack paths between any two KG entities via shortest-path with per-hop MITRE technique mapping, vulnerability lookup, and feasibility scoring. Compute 0-100 exposure scores from internet-facing services, known vulnerabilities, attack path count, and high-risk service types. Generate ranked reduction recommendations with specific shell commands for binding restrictions, firewall rules, service hardening, and disabling unnecessary services. Cross-references hardening assessment findings.

Explainable Autonomous Decisions

Full transparency for every autonomous and human-approved action. View the reasoning chain for any decision — trigger, context, options considered, chosen action, and justification. Browse decision history with time-windowed queries and summary statistics (autonomous vs human, action type breakdown, review status). Provide human feedback (approved/rejected/neutral) on past decisions for learning and audit. Internal record_decision() helper integrates with the orchestrator's tool execution path.

Security Posture Scoring

Composite 0-100 security posture score from six weighted dimensions: hardening compliance (25%), vulnerability exposure (20%), incident load (20%), config drift (15%), network exposure (10%), and patch freshness (10%). Letter grades A-F. Historical trend analysis with min/max/avg statistics. Drift detection against saved baselines with per-dimension comparison and severity classification (stable/degraded/improved). Snapshot baselines for regression tracking after hardening campaigns.

Contextual Risk Framing

Asset-centric risk scoring using Risk = (Threat x Vulnerability x Asset_Value) / Countermeasures, normalized 0-10. Threat scores from active incidents and tracked attackers. Vulnerability scores from surface scans and attack paths. Countermeasure scores from applied hardening, active deceptions, and detection rules. Prioritized risk register across all assets. Compliance mapping to CIS, NIST 800-53, PCI-DSS v4.0, HIPAA, and SOC2 with pre-seeded control mappings and live status refresh from hardening checks. Asset registry with criticality levels (critical/high/medium/low/informational) affecting risk weighting.

Learning From Incidents

Extract structured lessons from closed incidents: detection gaps, response delays, tool gaps, process improvements, and positive findings. MITRE ATT&CK technique extraction from incident text via keyword heuristics. Compute detection and response latencies. Find past incidents by indicator overlap across events, evidence, and honeypot attacks. Analyze historical playbook effectiveness with success/failure rates and average durations. Propose new detection rules (pattern, threshold, correlation, behavioral) from incident analysis — stored inactive for human review before activation.

Adaptive Playbooks

AI-composed custom playbooks from an objective and optional incident context. Rule-based step composition for containment, eradication, recovery, investigation, hardening, and full lifecycle workflows. Each step has a tool call, arguments, description, and on_failure policy (skip/abort/retry). Human adaptation of pending playbooks via feedback — modify, remove, or reclassify steps. Sequential execution with per-step results, failure handling, and automatic retry. Template library with success rates, failure counts, and average duration tracking. New templates auto-created from successful runs for future reuse.

Intelligence Event Subscribers

Reactive event bus subscribers wiring intelligence tools into the pipeline: posture auto-recalculation on hardening/incident changes, auto-extract lessons on incident closure, hypothesis suggestions on new evidence, attack surface rescans on network changes, indicator pattern matching on new IoC observations, playbook effectiveness tracking on completion, and detection rule evaluation against monitor/IDS alerts with effectiveness scoring.

Security Architecture (5)

Rust Core + Python Fallbacks

Security-critical components (audit logger, permission gate) have Rust implementations via PyO3. Equivalent Python fallbacks auto-detected at runtime for cross-platform development.

Scope-Locked Targeting

Offensive tools can only target systems within a user-defined scope whitelist. Scope is enforced by the permission gate before tool execution.

Rate Limiting

Per-tool rate limits with configurable per-minute caps. Prevents runaway automation and resource exhaustion from aggressive tool use.

Confirmation Requirements

Dangerous operations require explicit user confirmation even when autonomous mode is enabled. Lowering safeguard levels requires typed confirmation phrases.

2FA (TOTP)

Time-based one-time passwords required to leave FORTRESS mode. Prevents accidental or social-engineered de-escalation of safety constraints.

Deception Mode (4)

Honeypot Deployment

Deploy SSH, HTTP, SMB, and FTP honeypots with three interaction levels: Low (banner only, log attempts), Medium (accept weak creds, fake responses), High (full shell/filesystem, capture everything). Each honeypot runs in an isolated rootless Podman container with resource limits. Simulated mode available for development/testing without containers.

Attack Monitoring

Real-time tracking of attacker activity: connection events, authentication attempts, commands executed, files uploaded, and exploit attempts. SQLite-backed session and event database. Credential capture with frequency analysis. Aggregate statistics per honeypot with event type breakdown.

Attacker Analysis

MITRE ATT&CK technique mapping from observed attacker behavior. Session analysis with risk scoring (low/medium/high/critical). Attacker profiling across all honeypots — timing patterns, techniques used, honeypots targeted. Attack timelines with hourly bucketing for pattern detection.

Intelligence Extraction

Extract IoCs from honeypot data: attacker IPs, captured credentials, URLs, commands, user agents, file hashes. Generate threat intelligence reports (JSON or text). Check attacker IPs against reputation feeds. Export findings to investigation cases (Phase 3 evidence system integration).

Warrior Mode (Bug Hunter) (7)

Scope Management

Define and enforce target whitelists for authorized engagements. CIDR-aware scope checking (IP ranges, wildcard domains, URLs). SQLite-backed scope persistence with full audit trail of scope changes. Every offensive tool validates targets against the active scope before execution.

Reconnaissance

Scope-locked network reconnaissance: port scanning (nmap or pure-Python connect-scan fallback), service and version detection, host discovery in CIDR ranges, DNS enumeration (A/AAAA/MX/NS/TXT/CNAME/SOA), and service banner grabbing. All targets checked against whitelist.

Vulnerability Scanning

Nuclei-powered vulnerability scanning with severity filtering. SSL/TLS configuration analysis (certificate details, protocol support, cipher suites). Web technology fingerprinting (servers, frameworks, CMS, JS libraries). CVE database search via NIST NVD API. Directory and file discovery (gobuster or built-in httpx brute force).

Web Application Testing

HTTP method testing (detect dangerous TRACE/PUT/DELETE). Security header audit with grading (CSP, HSTS, X-Frame-Options, etc.). Parameter fuzzing with anomaly detection. SQL injection testing with safe error-based payloads. Reflected XSS testing with harmless probe tags. All tests use non-destructive detection techniques.

API Security Testing

12-tool advanced API testing suite: GraphQL introspection, injection, and depth-limit testing. JWT analysis — decode tokens, test alg:none bypass, RS256-to-HS256 algorithm confusion, and weak-secret brute-force. OAuth redirect_uri bypass, PKCE enforcement, and state/CSRF validation. REST verb tampering and rate-limit detection. All tests use non-destructive detection techniques.

Exploit Research

Search public exploit databases (Exploit-DB, NVD references) by keyword, CVE ID, or product name. Retrieve exploit metadata and details. Research only — no exploit code execution. No scope required for information lookups.

Findings & Reporting

SQLite-backed vulnerability findings database with severity classification (critical/high/medium/low/info), CVSS scoring, and CVE linking. Professional pentest report generation (JSON or text format) with executive summary, methodology, and recommendations. Responsible disclosure timeline tracking. Export findings as JSON, CSV, or Markdown.

Incident Commander Mode (5)

Incident Response (Tier 1)

Full IR case management: create and track incidents through containment, eradication, recovery, and closure. Collect evidence with SHA-256 hashing. Build incident timelines from events, evidence, and playbook runs. Execute automated IR playbooks (containment, eradication, recovery, full lifecycle). Export reports as JSON or formatted text.

Hardening Enforcement (Tier 2)

Apply hardening fixes identified by assessments — SSH config (root login, password auth), kernel parameters (IP forwarding, SYN cookies), filesystem permissions (sticky bits). Every change backs up the original and records an undo command. Batch application with snapshot groups for rollback. Verify fixes are still in effect after changes.

Counter-Intelligence (Tier 3)

Track and profile attackers with risk scoring and MITRE ATT&CK kill chain mapping. Deploy targeted deception assets (fake credentials, breadcrumbs, canary tokens, honey files, decoy services). Disrupt attacks with active countermeasures (C2 sinkholing, tarpitting, traffic redirection, throttling, lateral movement blocking). Offensive actions gated by a separate CI toggle.

Threat Assessment

Compute 0-10 threat scores from open incidents, tracked attackers, IDS alerts, and system indicators. Auto-suggest counter-intel activation when threat level is critical. Recommend safeguard level escalation or de-escalation based on posture. Historical trend analysis.

Counter-Intel Safety Toggle

Offensive CI actions (deception, disruption) require a separate runtime toggle — default OFF every session. Even at Level 1 with full autonomy, offensive actions are blocked unless explicitly enabled with an exact acknowledgment phrase. Read-only CI (tracking, profiling) works without it.

Rust Core Acceleration (Phase 11C) (5)

CryptoEngine

Full cryptographic suite in Rust via PyO3: Ed25519 keypair generation (ed25519-dalek, OsRng CSPRNG), Ed25519 signing (64-byte signatures) and stateless verification, AES-256-GCM authenticated encryption (12-byte nonce prepended to ciphertext), Argon2id key derivation (m=64MB, t=3, p=4 — OWASP recommended), SHA-256 hashing, and cryptographically secure random byte generation. All key material (signing key, verifying key, AES key) zeroized on drop via the zeroize crate — prevents key recovery from memory. Python fallback uses cryptography library (AESGCM, Ed25519), argon2-cffi, and hashlib with best-effort bytearray zeroization in __del__. Identical API for seamless substitution.

SandboxEngine

Process isolation via fork() + unshare() with CLONE_NEWUTS (hostname) and CLONE_NEWNET (network namespace, optional based on network_access flag). Seccomp-bpf filtering applied per-tool via SyscallFilter profiles. Resource limits via cgroups v2: memory.max, cpu.max, pids.max — written to /sys/fs/cgroup/dagda/sandbox_{id}/. Configurable parameters: max_memory_mb (default 512), max_cpu_percent (50), max_pids (32), max_duration_secs (300), filesystem_read/write paths. Execution: fork child, redirect stdout/stderr to pipes, poll with 10ms granularity, enforce timeout via SIGKILL. Returns SandboxResult with exit_code, stdout, stderr, timed_out, resource_exceeded. Linux only — is_available() returns false elsewhere. Python fallback uses subprocess.run() with resource.setrlimit() (RLIMIT_AS, RLIMIT_NPROC) on Linux; is_available() honestly returns False (no real isolation).

SyscallFilter & Registry

Per-tool syscall whitelisting with 7 profiles. Minimal: 20 essential syscalls (read, write, close, mmap, etc.). FileReadOnly: Minimal + open/stat/getdents/lseek/readlink. FileReadWrite: FileReadOnly + unlink/rename/mkdir/chmod/fsync. NetworkClient: Minimal + socket/connect/sendto/recvfrom/poll/epoll. NetworkServer: NetworkClient + bind/listen/accept. ProcessSpawn: Minimal + FileReadOnly + fork/clone/execve/wait/pipe. Full: all syscalls (trusted tools only). SyscallRegistry maps 12 tool categories to profiles: system and defensive/monitor -> FileReadOnly, investigation/osint -> NetworkClient, honeypot -> ProcessSpawn, arena -> Full, plugin -> Minimal. Custom syscalls addable per-filter. Stubs used when Rust unavailable — no-op but same API.

IPC Layer

Length-prefixed JSON over Unix domain socket at /tmp/dagda-ipc.sock for high-performance Rust-to-Python communication on hot paths. 4-byte big-endian length prefix + JSON payload, max 10MB per message. Request format: {id, method, params}. Response format: {id, result, error}. Non-blocking accept loop with 50ms sleep. Graceful shutdown via atomic running flag. Currently supports ping and crypto.hash_sha256 methods. Python fallback routes method calls directly to Python functions (no socket) — is_connected() always True.

Rust/Python Bridge

Unified import layer in brain/native.py. Try/except pattern: attempts to import from dagda_core (compiled Rust PyO3 module), falls back to Python implementations on ImportError. All exports (CryptoEngine, SandboxEngine, SyscallFilter, SyscallRegistry, IpcServer, IpcClient, AuditLogger, PermissionGate) have identical APIs regardless of backend. is_rust_available() and get_backend_name() for runtime introspection. Python stubs are honest about capabilities — SandboxEngine.is_available() returns False, SyscallFilter methods are no-ops. Enables cross-platform development (Windows/macOS) while deploying with full Rust security on Linux.

Plugin System (Phase 11D) (5)

TOML Manifest & Lifecycle

Plugins defined by dagda-plugin.toml manifest specifying: plugin ID, name, version, min Dagda version, description, author, entry point (module:function). 5-state lifecycle: DISCOVERED (found on disk) -> VALIDATED (signature checked, permissions OK) -> ENABLED (API initialized, tools registered) -> DISABLED (was enabled, now inactive) -> ERROR (failed validation or runtime). PluginLoader scans configured plugin_dirs recursively for manifests. startup() auto-discovers, validates, and enables trusted plugins in sequence.

Plugin Signing & Trust

Deterministic SHA-256 digest computed from all plugin files: sorted by relative path for reproducibility, each file's path and contents hashed, __pycache__/.git excluded. Ed25519 signature (64 bytes, base64-encoded) stored in manifest [signature] section alongside the public key. Verification recomputes digest and validates against signature using CryptoEngine (Rust or Python). Trust determined by public key membership in configured trusted_keys list. Untrusted plugins rejected unless allow_untrusted=True in config.

Plugin Sandbox

3 isolation levels selected by trust status. IN_PROCESS: trusted plugins run in same Python process with PluginAPI restrictions only — minimal overhead. SUBPROCESS: untrusted plugins on non-Linux run in child process with timeout (default 300s) and resource.setrlimit() (RLIMIT_AS, RLIMIT_NPROC). RUST_SANDBOX: untrusted plugins on Linux run in full Rust SandboxEngine with namespace isolation (UTS, network), seccomp-bpf syscall filtering (Minimal profile by default), cgroups v2 resource limits (max_memory_mb, max_pids), and filesystem path restrictions. Selection is automatic: Rust sandbox preferred when available, subprocess as fallback.

Plugin API (Restricted Facade)

Plugins never get direct access to Dagda internals — PluginAPI enforces per-plugin permissions declared in the manifest. Event bus: subscribe/publish restricted to declared prefixes via prefix matching (PermissionError on violation). Knowledge graph: gated by permission level — 'none' blocks all, 'read' allows kg_find_entities(), 'write' adds kg_add_entity(). Scoped tool registration: plugins can register tools that appear in the tool registry. cleanup() on disable unsubscribes all event handles. Permissions dataclass includes: modes, min_safeguard, network_access, filesystem_read/write paths, event_bus_subscribe/publish prefixes, knowledge_graph level, max_memory_mb.

Companion Svelte web component provides the same views with interactive Crow selection, geographic/logical topology visualization, and murmuration control (launch, pause, cancel). Both interfaces update via WebSocket event stream.

12 Fleet Tools

flock_status: overview of all enrolled Crows with health and capability summary. crow_enroll: register a new Crow agent with identity verification. crow_dismiss: remove a Crow from the fleet with cleanup. crow_details: deep inspection of a specific Crow (health, tasks, history). murmuration_scan: launch distributed scanning across selected Crows. murmuration_monitor: coordinated monitoring operation. murmuration_defend: fleet-wide synchronized defense action. murmuration_collect: parallel artifact collection from multiple endpoints. murmuration_deploy: coordinated deception asset deployment. murmuration_harden: rolling fleet hardening with verification. broadcast_ioc: push IOCs to all Crows for immediate detection. fleet_report: generate comprehensive fleet security posture report.

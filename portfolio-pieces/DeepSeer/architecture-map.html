01

System Architecture Overview

Deep Seer has three main parts that work together:

  • The Website (Client) — What users see and interact with. It runs a 3D globe, shows data layers (flights, ships, earthquakes, etc.), and lets users build investigations. Built with React and rendered in the browser.
  • The Server (Backend) — The brain. It fetches data from 90+ sources, manages user accounts, runs AI analysis, talks to the database, and pushes real-time updates to the browser over WebSockets.
  • The Database (PostgreSQL) — Long-term memory. Stores users, investigations, evidence, audit logs, and cached data. Uses PostGIS for geographic queries.

They communicate like this: the browser sends HTTP requests and WebSocket messages to the server. The server reads/writes the database and fetches from external APIs. Data flows back to the browser in real time.

Scale: The codebase has 98 server adapters, 94 services, 106 panels, 70 stores, 106 globe layers, and 35 database models. It is a full-featured OSINT intelligence platform.

Build Pipeline

# Development (Vite HMR + nodemon)
npm run dev          # concurrently: client (5173) + server (3001)

# Production build order
npx tsc --build shared/tsconfig.json --force   # 1. Shared types → dist/
npm run build -w client                         # 2. Vite bundle → client/dist/
npm run build -w server                         # 3. tsc → server/dist/

Communication

  • HTTP: Vite dev proxy forwards /api/* and /ws to localhost:3001
  • WebSocket: Persistent connection for real-time entity updates (flights, ships, alerts)
  • Database: Prisma ORM over TCP to PostgreSQL (connection pooling via Prisma)

Key Technical Decisions

  • TypeScript throughout (strict mode, shared types package)
  • CesiumJS assets served from node_modules/cesium/Build/ (not bundled)
  • SGP4 satellite propagation runs client-side (no server load)
  • Zustand for state management (no Redux boilerplate)
  • Static class methods always wrapped in arrow functions (context safety)

The app runs on Render, a cloud hosting service (like a simpler AWS). It is deployed as a Docker container in the Oregon (US West) region.

Database (PostgreSQL + PostGIS)

The database is PostgreSQL — an industry-standard relational database. It stores everything the app needs to remember: user accounts, investigations, evidence files, audit logs, and cached data.

It also has PostGIS, an extension that lets it understand geography (e.g., "find all evidence within 5 miles of this point").

The database has 35 tables (called "models") organized into groups:

  • Users & Auth: User accounts, API keys, refresh tokens, subscriptions
  • Investigations: Cases, members, evidence, annotations, manual links, timeline events
  • Evidence Integrity: Artifacts, provenance chains, SHA-256 hashes, timestamps
  • AI: Chat history, AI outputs, narrative briefings, usage/overage tracking
  • Analysis: Hypotheses, tests, evidence links, analyst actions, bias alerts
  • Operational: Audit logs, reports, documents, watchlists, workflows, saved queries

Cache Layer (Redis / In-Memory)

Redis is a super-fast in-memory data store used for three things:

  • Rate Limiting: Prevents users from hammering the API too fast
  • Caching: Stores frequently-requested API responses so the server doesn't re-fetch them
  • Session State: Tracks active WebSocket connections and their subscriptions

If no Redis server is available (e.g., in local development), everything falls back to in-memory storage automatically. The app works fine without Redis — it just can't share cache across multiple server instances.

Providers
Upstash (serverless) or Render Redis
Cost
$0 (in-memory fallback) → $10-20/mo (managed Redis)

Configuration

REDIS_URL=redis://default:password@host:6379

Fallback Strategy

The server checks for REDIS_URL at startup. If absent or connection fails, it uses a Map<string, {value, expiry}> in-memory cache with identical API. This means:

  • Single-instance deployments work without Redis
  • Multi-instance (horizontal scaling) requires Redis for shared state
  • Rate limits reset on server restart when using in-memory

AI Services (Anthropic Claude)

Deep Seer uses Anthropic's Claude AI for intelligent analysis. Think of it as an AI analyst that can read data, summarize situations, and answer questions about investigations.

It powers:

  • Chat analysis: Ask questions about your investigation data in natural language

Two AI models are used: Haiku (fast and cheap, for simple tasks) and Sonnet (smarter, for reasoning-heavy analysis). Users get daily token quotas based on their subscription tier.

Error Tracking (Sentry)

Sentry catches errors in both the website and server before users report them. When something crashes, Sentry captures a full trace (what happened, where, and why) and sends an alert.

It tracks both the browser (client-side JavaScript errors) and the server (Node.js exceptions, unhandled promise rejections). Optional — the app works without it, you just won't get automatic error alerts.

3D Globe (CesiumJS + Cesium Ion)

The centerpiece of Deep Seer is a 3D globe powered by CesiumJS — the same technology used by NASA, NOAA, and the US military for geospatial visualization.

The globe renders satellite imagery, terrain, 3D buildings, and all the data layers (flights, ships, cameras, environmental data). Cesium Ion is a companion cloud service that provides high-resolution terrain data and satellite imagery tiles.

The app also includes custom visual effects (shaders): CRT monitor, Night Vision, FLIR thermal, Matrix rain, Anime style, Surveillance mode, and Crisis mode.

Client-Side Storage

The browser stores some data locally so the app feels fast and works partially offline:

  • localStorage: User preferences (theme, active layers, panel positions, last-used investigation)
  • IndexedDB: Larger datasets cached for offline use (entity histories, downloaded evidence)
  • Service Worker: Caches API responses and map tiles so previously viewed areas load instantly

All local data is encrypted at rest using the user's session key. Clearing browser data resets preferences but doesn't affect server-side data.

Authentication & Security

Deep Seer uses industry-standard security practices:

  • Passwords: Never stored in plain text. Hashed with bcrypt (12 rounds) so even if the database leaks, passwords are safe.
  • Login tokens: Short-lived (15 minutes), automatically refreshed. If a token is stolen, it expires quickly.
  • Evidence integrity: Every piece of evidence gets a SHA-256 hash and HMAC signature. Can't be tampered with without detection. Optionally timestamped via OpenTimestamps for legal admissibility.
  • Rate limiting: Prevents brute force attacks and API abuse.
  • CORS & CSP: Browser security headers prevent cross-site attacks.

Evidence Integrity Pipeline

  1. SHA-256 hash of artifact content at capture time
  2. HMAC signature using EVIDENCE_SIGNING_KEY for chain-of-custody
  3. RFC 3161 timestamps (optional) via trusted TSA
  4. OpenTimestamps (optional) for blockchain-anchored proof
  5. Provenance graph (ProvenanceNode + ProvenanceEdge) tracks all transformations

Environment Variable Reference

Environment variables are settings that tell the app how to connect to external services. They are stored in .env files (one for the client, one for the server) and never committed to Git.

Below is every environment variable the app uses, what it does, and where to get the value.

Client vars go in client/.env (must be prefixed VITE_). Server vars go in server/.env. See .env.example at project root.

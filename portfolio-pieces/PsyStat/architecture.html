PsyStat Nexus — Architecture & Services Map

Complete technical reference for developers and maintainers

1 Architecture Diagram

How the pieces of PsyStat connect: users access the app from any device, the app talks to two backend services (Supabase for user accounts and data, Railway for running statistics and AI).

Three-tier architecture: Expo/React Native universal client communicates via REST to a FastAPI backend on Railway and directly to Supabase (auth, Postgres, storage). The backend calls the Anthropic API for AI features. JWT tokens from Supabase authenticate all protected requests.

2 Services & Accounts

S
SupabaseRequired

Handles user sign-up/login, stores all user data (analyses, notes, projects), and enforces per-user data isolation so nobody can see another user's work.

PostgreSQL 15 via Supabase-hosted instance. Auth uses email/password with JWT (HS256). Row-level security on all 7 tables. Client SDK communicates directly from frontend; backend uses service-role key for admin operations.

R
RailwayRequired

Runs the Python backend that performs all statistical calculations (t-tests, ANOVA, regressions, etc.) and routes AI requests to Claude. This is where the heavy math happens.

FastAPI + Uvicorn on Railway. 77 routers, 240+ endpoints. Dependencies: scipy, statsmodels, pandas, numpy, scikit-learn, networkx, lifelines, anthropic SDK. SQLite-backed rate limiting (60 req/min auth, 20 unauth). NumpySafeResponse serializer for numpy types.

3 Database Schema

Supabase stores 7 tables. Every table is locked down so each user can only see and edit their own data. All tables use auto-generated UUIDs and timestamps.

PostgreSQL 15 on Supabase. All tables have RLS enabled with policy auth.uid() = user_id. Tables with updated_at have a trigger function (update_updated_at()) that auto-updates on row modification. Foreign keys reference auth.users.

Relationships: Every table connects to Supabase's built-in auth.users table through a user_id column. The annotations table can point to any analysis via its target_id. The result_graph stores a network of how analyses relate to each other.

Relationships: All FK references point to auth.users(id). annotations.target_id is a soft reference (text, not FK) to any entity UUID. result_graph stores a full adjacency list (nodes + edges as JSONB arrays) per user. Indexes on user_id, module, and target_id columns.

4 API Endpoints

The backend has 77 router groups with 240+ endpoints. All statistical computations, AI features, and data operations go through these. Most accept POST requests with JSON data.

FastAPI app with 77 registered routers. Default response class: NumpySafeResponse. Auth: JWT middleware with public, optional-auth, and protected path tiers. Rate limiting: 60 req/min (auth), 20 req/min (unauth). Max body: 10 MB, max array: 100K items, max nesting: 10.

6 File Structure

How the project is organized on disk. The app code and backend are in separate directories, with marketing pages at the root level.

Monorepo structure. Frontend is an Expo managed workflow project. Backend is a standalone FastAPI app deployed to Railway. Static HTML pages are deployed to Vercel alongside the Expo web build.
